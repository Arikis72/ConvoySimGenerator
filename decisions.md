## 2026-05-29 (evening) — max_events constraint: velocity events count toward total

Decision:
`max_events` is a hard cap on the TOTAL number of events per scenario, including:
- Velocity waypoints (from the velocity profile)
- Loss/Resume pairs (2 events each)
- FORT event (1 event if present)

When calculating how many loss pairs can fit, the algorithm:
1. Generates the velocity profile and counts its waypoints
2. Calculates remaining budget: `budget = max_events - velocity_count`
3. Limits loss pairs to `budget // 2` (since each pair = 2 events)

Reason:
The user explicitly stated that `max_events=3` was violated when scenarios contained 12 total events (8 velocity + 2 loss + 2 resume). The intent is to limit the TOTAL complexity of each scenario, not just discrete loss events. Velocity waypoints contribute to CSV row count and simulation complexity, so they must be budgeted. This ensures generated scenarios match user expectations and conforms to the specified constraint.

Alternatives considered:
1. Treat velocity waypoints separately (only cap loss events) — rejected because it violates the semantic meaning of `max_events` (total events).
2. Cap velocity profile generation — rejected because velocity profiles are generated by type (nV=1-2 points, mV=8-12, hV=6-10, hB=3-4 points) and reducing them would break the velocity patterns (e.g. hard brake needs at least 2 points for deceleration).
3. Post-filter velocity events after loss placement — rejected because velocity profile waypoints form an interpolated curve; removing points silently corrupts the curve (breaks hard-brake deceleration, flat-lines variable profiles).

Consequences:
- Scenarios with variable velocity profiles (8-12 velocity points) can have 0-1 loss pairs with max_events=10
- Scenarios with nominal velocity profiles (1-2 velocity points) can have 0-4 loss pairs with max_events=10
- User can adjust max_events up (e.g., 15, 20) to allow more complex scenarios
- The constraint is now truly enforced and predictable

## 2026-05-29 — Pause button: sleep-loop in progress callback

Decision:
The Pause button uses a threading.Event that is checked in the progress callback.
When paused, the callback enters a loop that sleeps 100ms between checks, breaking
when the event is cleared (user resumes) or the stop signal is set (user aborts).
The button label toggles between "Pause" and "Resume" to show current state.

Reason:
Sleep-loop is simple and non-blocking. 100ms sleep means responsive pause (user sees
pause within one scenario generation cycle) while minimizing CPU use. The progress
callback is called once per scenario, so pause checks happen frequently enough.
The button label provides clear visual feedback of state.

Alternatives considered:
Condition variables with notify (rejected — more complex, no benefit over Event).
Separate pause thread (rejected — unnecessary overhead; event is sufficient).
Blocking in generator code (rejected — tightly couples pause logic to generation).

Consequences:
Pause takes effect at the next progress callback (next scenario), not instantly
mid-scenario. Users cannot pause during a single scenario's generation. This is
acceptable for scenarios that complete in milliseconds.

## 2026-05-29 — Stop button implementation via threading.Event

Decision:
The Stop button uses a `threading.Event` that is checked in the progress callback
(invoked once per scenario generated). When the user clicks Stop, the event is set,
causing the next progress callback to raise a RuntimeError with message
"Generation stopped by user". The generation thread catches this, cleans up,
and returns gracefully.

Reason:
Threading.Event is the standard Python pattern for inter-thread signalling. Checking
in the progress callback ensures responsive cancellation (checked once per scenario)
without constant polling overhead. The exception-based control flow is explicit and
the exception message identifies user cancellation vs. generation errors.

Alternatives considered:
Polling in the main generation loop (rejected — tight coupling to generator logic;
harder to maintain). Using a flag with busy-waiting (rejected — wastes CPU).
Forcibly terminating the thread (rejected — unsafe; can leak resources).

Consequences:
Users can abort generation but not mid-scenario (generation of a single scenario
completes before checking the stop signal). Partially-generated scenarios are
discarded, not exported. This is acceptable for the use case.

## 2026-05-29 — Category filtering format and logic

Decision:
Category filters in `selected_categories` use the format `category_type:value`, e.g.,
`velocity_type:hV`, `gap_type:sG`, `loss_type:qs`, `fort:yes`, `fort:no`.

Multiple filters for the same category type are combined with OR logic (e.g., selecting
both `velocity_type:nV` and `velocity_type:hV` includes scenarios with either type).
Unchecked filters default to "include all" for that category (non-restrictive).

Reason:
This format is explicit and self-documenting in code. The OR logic mirrors user
expectations: "I want these *or* these scenarios". Non-restrictive defaults ensure
that unchecking all boxes for a category behaves intuitively (all become possible,
as if no filter is applied).

Alternatives considered:
A single `include_velocity_types: List[VelocityType]` field per category (rejected —
would require multiple fields, cluttering ScenarioConfig; the string list is more
flexible and compact).

Consequences:
The generator's `_filter_by_category()` method applies OR logic across selected
values for each category type. The GUI collects checkboxes and builds the
`selected_categories` list. Both generator and GUI validate this format consistently.

## 2026-05-29 — Immediate requirement tracking (not deferred)

Decision:
Any new requirement or change stated during a conversation must be added to
REQUIREMENTS.md (with status ⏳) before code is written, and updated to ✅
immediately after verification — within the same response, not at session end.

Reason:
Deferred batch updates get skipped when sessions run long or context fills.
Requirement 2.2 was partially implemented while documentation claimed full
completion precisely because tracking happened after the fact.

Alternatives considered:
End-of-session housekeeping pass (the previous approach — caused req 2.2 slip).

Consequences:
REQUIREMENTS.md reflects reality at every moment during a session.
The status table is the authoritative signal for what is truly done.

## 2026-05-29 — Doc consolidation: 8 files, one purpose each

Decision:
Reduce the MD file set from 15 to 8. Each file has a single stated
responsibility. No new MD files should be created; content that doesn't
fit belongs in the nearest existing file.

Surviving files: README.md, STATUS.md, progress.md, decisions.md,
REQUIREMENTS.md, TASKS.md, PLANNING.md, CLAUDE.md.

Deleted: COMPLETION_STATUS.md, PROJECT_SUMMARY.md, INDEX.md,
ARCHITECTURE.md, QUICKSTART.md.

Reason:
Having 4+ files describing "what is done" caused inevitable drift.
Updating one file left others stale, giving Claude false confidence
that requirements were complete.

Alternatives considered:
Keeping all files with strict "last updated" dates (rejected — still
requires updating multiple files for every change).

Consequences:
Every question about project state has one canonical answer file.
Claude session-start reads exactly 5 files (README, STATUS, progress,
decisions, REQUIREMENTS) and gets a complete picture.

## 2026-05-29 — Scene_Gen root field: always append timestamp subfolder

Decision:
The "Output Root Folder" GUI field (and `--output` CLI flag) specifies
the Scene_Gen root. Every run appends a `ddmmyyyy_hhmm` subfolder
automatically, regardless of whether the user types a path or leaves the
field blank.

Reason:
The original code only appended the timestamp when the field was blank.
If the user browsed to a folder, CSVs went directly there — successive
runs overwrote each other and requirement 2.2 was violated.

Alternatives considered:
Using a fixed "Scene_Gen" folder name alongside any user-supplied path
(rejected as confusing — user's chosen name IS the Scene_Gen root).

Consequences:
GUI label updated to "Output Root Folder". Help text explains the
automatic subfolder. Behaviour is now consistent with requirement 2.2.

## 2026-05-29 — Single shared loss type per scenario (Bug 12 fix)

Decision:
All loss/resume pairs within a single scenario share one randomly chosen
LossType. The generator picks `shared_loss_type = random.choice(list(LossType))`
once and applies it to every pair.

Reason:
The naming encoder uses only the first loss type to label all pairs in the
filename (e.g., `idL2qs` = 2 losses, both quick-short). If pairs had
different types, the filename was misleading. Enforcing one shared type
makes the name accurate.

Alternatives considered:
Encoding mixed types in the filename (e.g., `idL1qs1sl`) — rejected
because it complicates naming, parsing, and the 30-char limit.

Consequences:
Scenarios with mixed loss types are not generated. This is a known
limitation listed in TASKS.md § Deferred.

## 2026-05-29 — _assign_timestamps returns actual_loss_types (Bug 11 fix)

Decision:
`_assign_timestamps()` returns a 4-tuple:
`(velocity_events, loss_resume_pairs, fort_event, actual_loss_types)`
where `actual_loss_types` reflects only the pairs that were successfully
placed (some may be dropped if no valid slot exists).

Reason:
The scenario name was generated from the originally planned event count,
not what was actually placed. This caused names like `_1ES` for scenarios
where FORT was silently dropped because the velocity profile ended exactly
at scenario_duration_s, leaving no room.

Alternatives considered:
Post-hoc inspection of the returned pairs/fort_event to derive types
(equivalent, but less explicit — the 4-tuple makes the contract clear).

Consequences:
All 5 tests in TestAssignTimestamps updated to unpack 4 values.
Scenario names now always accurately reflect CSV content.

## 2026-05-29 — FORT tail-clearing algorithm (Bug 11 fix, part 2)

Decision:
When a FORT event is planned, `_assign_timestamps()` removes velocity
events from the tail of the profile until the last remaining velocity
event is at least `min_event_separation_s` before `scenario_duration_s`.
This guarantees a valid slot for FORT exists.

Reason:
Velocity profiles end exactly at `scenario_duration_s`. FORT must come
after all other events with at least `min_sep` clearance — so with no
tail clearing, FORT could never be placed and was always silently dropped.

Alternatives considered:
Shortening the velocity profile during generation instead of post-hoc
trimming (rejected — the profile generation and timestamp assignment are
intentionally separate concerns).

Consequences:
Scenarios with FORT events may have slightly fewer tail velocity events.
The trade-off is acceptable: FORT placement is reliable and the name
is accurate.

## 2026-05-29 — Initial gaps: one value per CSV column (Bug 10 fix)

Decision:
`format_initial_gaps_row()` emits a 6-field comma-separated row:
`Initial gaps:,{gap1},{gap2},,,` — label in column 0, each gap in its
own column, remaining columns empty.

Reason:
The original implementation concatenated all gaps into a single string
in column 0: `Initial gaps:,10 10,,,`.  When opened in a spreadsheet,
gaps did not align with the data columns below them, breaking readability
and ConvoySIM parsing.

Alternatives considered:
Semicolon-separated values inside one cell (rejected — not standard CSV).

Consequences:
Tests updated to verify the 6-field format. ConvoySIM Stage A can now
parse the gaps row by column index.

## 2026-05-29 — GUI redesigned to 2 tabs from 4

Decision:
Merged the original 4-tab GUI (Parameters, Generation, Browser, Viewer)
into 2 tabs: "Generate" (parameters + progress + log) and "Scenarios"
(list | details | editable CSV table).

Reason:
User request: fewer tabs reduces navigation friction. The viewer and
browser are combined because selecting a scenario and viewing its CSV
are the same workflow.

Alternatives considered:
3 tabs (Generate | Browse | Edit) — rejected as still one tab too many.

Consequences:
README.md updated to remove references to 4 tabs.
The combined Scenarios tab uses a vertical PanedWindow (top: list+details;
bottom: editable CSV table).

## 2026-05-28 — Use CSV format matching ConvoySIM Stage A

Decision:
Generated scenarios must exactly match ConvoySIM Stage A scenario_timeline.csv
format: 6 columns, specific header names, Row 1 = description, Row 2 = gaps.

Reason:
Zero integration friction — generated scenarios load directly into ConvoySIM
without translation or additional validation.

Alternatives considered:
Intermediate format with conversion step; flexible column structure.

Consequences:
csv_writer.py is tightly coupled to the ConvoySIM column spec. Any
ConvoySIM format change requires a matching update here.

---

## 2026-05-28 — Separate scenario generation from simulation

Decision:
SimGenerator is a standalone companion tool to ConvoySIM, not a built-in
feature of the simulator.

Reason:
Generation has distinct concerns (randomisation, naming, category selection,
UI) separate from simulation (physics, state machines, visualisation).
Keeping them separate allows independent development and testing.

Alternatives considered:
Integrating generation into ConvoySIM's bulk simulation system; making it
a utility module inside the main package.

Consequences:
SimGenerator has its own project structure, documentation, and test suite.
Integration with ConvoySIM is through the standard CSV format only.
